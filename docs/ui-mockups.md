```
│                                          < 1  2  3  ... 25 >          │

Статусы:
  ⏳ queued    — в очереди
  ● running    — выполняется (с progress bar + текущая страница + счётчик)
  ✓ done       — готово (зелёный)
  ✕ cancelled  — отменено (серый)
  ⚠ failed     — ошибка (красный, показ error_message, кнопка "повторить")

Кнопка ✕ у задачи — cancel (revoke Celery + status=cancelled).
Блок задач схлопывается, если все завершены (автоматически скрыть через 30 сек).
```

### Что происходит при клике "Начать парсинг":
1. Backend: `POST /companies/{id}/scrape-tasks/start`
2. Для каждой enabled ячейки матрицы (category × region) создаётся fi_scrape_tasks row
3. Для каждой задачи — `scrape_offers_task.delay(task_id)` в Celery
4. Frontend сразу показывает задачи в "Активные задачи" со статусом queued
5. Polling `GET /companies/{id}/scrape-tasks?active=true` каждые 2 сек
6. Celery worker забирает задачу → status=running, прогресс обновляется
7. Завершение → status=done/failed, через 30 сек блок может свернуться

## 2. Форма маппинга категории (общая для ScrapeConfigPanel и Wizard)

Текущее состояние: wizard и panel имеют **разные формы**. Надо унифицировать.

### Новая форма — добавление/редактирование category mapping

```
┌─ Добавить маппинг категории ─────────────────────────┐
│                                                       │
│  Категория ритейлера *                                │
│  ┌────────────────────────────────────────────────┐  │
│  │ [Диваны и кресла                          ▾ ] │  │
│  │                                                │  │
│  │  Найдено 2 совпадения:                         │  │
│  │  • Диваны и кресла (раздел: Гостиная)         │  │
│  │  • Диваны угловые (раздел: Гостиная)          │  │
│  │                                                │  │
│  │  + Создать новую: "Диваны и кресла"           │  │
│  └────────────────────────────────────────────────┘  │
│                                                       │
│  URL на сайте (автозаполнение из справочника)         │
│  ┌────────────────────────────────────────────────┐  │
│  │ /catalog/gostinaya/divany/                     │  │
│  └────────────────────────────────────────────────┘  │
│                                                       │
│  Наша категория (необязательно)                       │
│  ┌────────────────────────────────────────────────┐  │
│  │ [— не указано —                           ▾ ] │  │
│  └────────────────────────────────────────────────┘  │
│                                                       │
│                            [Отмена]  [Сохранить]      │
└───────────────────────────────────────────────────────┘

Правила:
  ✓ Категория ритейлера — combobox:
    - Dropdown из fi_retailer_categories (per company)
    - Группировка по parent (раздел → подкатегория)
    - Можно создать новую — попадает в справочник
  ✓ URL подставляется из выбранной retailer_category, редактируемое
  ✓ Наша категория — необязательное (optional маппинг)
  ✗ Без retailer_category сохранить нельзя (primary signal для парсинга)
```

```
  ✓ Раздел (level 1) — name, опциональный URL
  ✓ Подкатегория (level 2) — name, обязательный URL, parent=раздел
  ✗ Дубликаты name в одном parent запрещены
```

## Вопросы для обсуждения

1. **Вкладка "Категории ритейлера"** — делать внутри ScrapeConfigPanel или отдельной секцией на странице?
   → Рекомендую: внутри ScrapeConfigPanel (логически связано)

2. **Auto-create retailer_category при создании mapping?** — Если в форме маппинга
   ввели новое название (не из dropdown), создавать автоматически в справочнике?
   → Рекомендую: да (combobox-поведение)

3. **Концepts duplications** — текущие таблицы fi_categories (наши) и fi_retailer_categories
   (ритейлера) похожи. Может использовать одну с company_id nullable?
   → Рекомендую: разные таблицы (семантически разные справочники, проще миграция)

4. **OfferForm — удалить полностью?** — Если "Добавить" убираем, OfferForm не нужен.
   → Рекомендую: удалить файл

5. **Bulk recategorize** — убираем кнопку, но оставить бэкенд endpoint?
   → Рекомендую: endpoint оставить (может вызываться автоматически после парсинга)

6. **Cancel задач** — hard stop невозможен мгновенно (Firecrawl запрос в процессе).
   Как обрабатывать?
   → Рекомендую: task проверяет `fi_scrape_tasks.status == cancelled` перед каждой страницей,
     прерывается мягко
